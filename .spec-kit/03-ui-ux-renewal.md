# UI/UX 전체 디자인 리뉴얼 사양

## 개요

- **목적**: AI SEO Blog Generator 서비스의 전체 UI/UX를 최신 웹 디자인 트렌드에 맞춰 단조로운(모노크롬·미니멀) 디자인으로 강력 개편
- **범위**: 디자인 시스템(토큰), 레이아웃, 네비게이션, 폼, 카드, 결과 영역, 공통 스타일
- **대상 사용자**: 콘텐츠 제작자, 마케터, 관리자

## 트렌드 리서치 요약 (2024–2026)

- **미니멀/모노크롬**: 단일 톤(hue)과 tints/shades 기반, 고대비·집중된 구성
- **디지털 피로 완화**: 색과다 자극 완화, 여백·타이포그래피 중심
- **접근성**: 고대비, 일관된 톤, 가독성 우선
- **Linear/Intentional**: 목적 있는 레이아웃, 간격 토큰, 명확한 타입 스케일
- **성능·코드**: 시각 자산 최소화, CSS 변수 기반 토큰

## 요구사항

### 기능 요구사항

- FR-001: 디자인 토큰을 모노크롬(흑·백·그레이 스케일)로 재정의
- FR-002: 액센트 컬러는 1종만 허용 — primary CTA 및 핵심 인터랙티브 요소에만 사용 가능한, **숫자로 정의된 채도 범위**(HSL saturation 10–20%)를 갖는 단일 액센트만 허용. 이 단일 액센트는 아래 “모든 UI 색상 saturation ≤5%” 검증 규칙에서 **예외**로 둔다.
- FR-003: 히어로·카드·폼·네비게이션에 동일 디자인 언어 적용
- FR-004: 상태(성공/경고/에러)는 그레이 스케일 내에서 표현하되, **색만으로 구분하지 않는다**. 상태 표시는 반드시 **다음 중 최소 2가지**를 사용해야 함: (1) **구분 가능한 아이콘**, (2) **명시적 텍스트 라벨**, (3) **비색각 시각 처리** — 테두리 스타일/패턴, 그림자, 투명도(opacity) 등. *NFR-001 및 WCAG 2.1 SC 1.4.1(Use of Color)과 정합: 색에만 의존하지 않고 위 방식 중 2가지 이상을 쓰면 의도 전달과 접근성 대안을 충족한다.*

#### 액센트 컬러 사용 가이드 (FR-002)

- **채도 범위**: 허용되는 단일 액센트는 **HSL saturation 10–20%**로 정의한다. 5% 이하는 그레이에 가깝고, 20% 초과는 “단일 desaturated 액센트” 원칙과 맞지 않으므로 이 범위로 제한한다.
- **예외 규칙**: “모든 UI 색상 saturation ≤5%” 검증은 **단일 허용 액센트를 제외**하고 적용한다. 즉, **단일 액센트를 제외한 모든 UI 색상은 saturation ≤5%**여야 한다.
- **사용 허용**: primary CTA(예: "생성하기", "저장", "제출"), 핵심 인터랙티브 요소(예: 주요 폼 제출 버튼, 한 페이지당 1~2개 수준의 핵심 액션)에만 위 범위의 단일 액센트 적용.
- **사용 금지**: 세컨더리 UI에는 사용하지 않음 — 세컨더리 버튼, 링크, 테두리, 배경, 장식 요소, 탭·네비게이션 일반 항목, 호버/포커스 링 등은 그레이 스케일로 유지.

#### 액센트 접근성 제약

- **대비율**: 액센트를 사용할 때 텍스트/아이콘은 배경 대비 **최소 4.5:1**(일반 텍스트), **3:1**(큰 텍스트·UI 컴포넌트) 유지. WCAG 2.1 AA 기준 준수.
- **그레이 전용 대안**: 모노크롬만 유지하려면 primary CTA도 그레이로 처리 가능. 예: `background: #2d2d2d`, `color: #fff`(다크 버튼), 또는 `background: #e5e5e5`, `color: #1a1a1a`(라이트 버튼). 이 경우 버튼 형태·크기·위치·라벨로 시인성과 affordance를 확보.

### 비기능 요구사항

- **NFR-001**: 접근성(a11y) 유지 — **WCAG 2.1 Level AA 완전 준수** (모든 적용 가능한 성공 기준(SC) 충족). 구현 시 대비율(SC 1.4.3), 포커스 가시성(SC 2.4.7), 이름·역할·값(SC 4.1.2), 색 사용(SC 1.4.1) 등 적용 가능한 모든 SC를 충족해야 한다.
- NFR-002: 반응형 유지
- **NFR-003**: 기존 HTML 구조·클래스명 호환 유지. 단, **NFR-001**(WCAG 2.1 Level AA 완전 준수) 충족을 위해 필요한 HTML 구조·클래스명·마크업 변경은 허용된다. 구현자는 접근성 요구사항을 만족하기 위해 NFR-003에 따른 마크업 수정이 가능함을 인지한다.
- **NFR-004 (다크 모드, 필수)**: 다크 모드 지원은 **필수**이다. (1) **자동 감지**: `prefers-color-scheme` 미디어 쿼리로 시스템 설정에 따른 라이트/다크 자동 적용. (2) **수동 오버라이드**: 사용자가 선택한 모드를 **영구 저장**하는 수동 토글이 있어야 하며, 토글이 설정된 경우 자동 감지보다 우선 적용. (3) **대비 검증**: 다크 모드에서도 텍스트 대비율 ≥ 4.5:1, UI 컴포넌트 대비율 ≥ 3:1 유지(테스트·리뷰 시 NFR-004 기준으로 검증).

## 검증 기준 (측정 가능한 수용 기준)

### 색상

- [ ] **채도**: **단일 허용 액센트를 제외한** 모든 UI 색상의 HSL saturation ≤ 5%. 단일 액센트는 HSL saturation 10–20% 범위로만 허용. (자동 검증: CSS 변수 또는 스타일시트에서 색상 추출 후, 액센트로 지정된 1색은 10–20% 검증, 그 외 모든 색은 ≤5% 검증.) **예시**: 배경·텍스트·테두리·세컨더리 버튼은 `hsl(0, 0%, 20%)` 등 saturation 0% 권장; primary CTA만 `hsl(220, 15%, 45%)`처럼 saturation 10–20% 허용. **참고**: WCAG 2.1 명암 대비(1.4.3) 및 비색 의존(1.4.1)에 따라, 저채도 액센트 사용 시에도 텍스트/아이콘 대비율 4.5:1(일반), 3:1(큰 텍스트·UI)을 충족해야 하며, 이 기준으로 검증 가능하다.
- [ ] **그레이스케일 팔레트**: 색상은 RGB hex이며 R=G=B(그레이스케일)여야 한다. 각 채널이 동일한 그레이스케일 값(R=G=B)을 사용하며, 나열된 기준값 V 중 하나에 대해 각 채널이 V±1(십진수, 즉 0–255 스케일에서 ±1 = ±0x01) 범위에 있어야 한다. 즉 허용 조건: R=G=B이고, 나열된 기준값 V 중 하나에 대해 |R−V| ≤ 1. 기준 hex: `#000000`, `#111111`, `#222222`, `#333333`, `#444444`, `#555555`, `#666666`, `#777777`, `#888888`, `#999999`, `#AAAAAA`, `#BBBBBB`, `#CCCCCC`, `#DDDDDD`, `#EEEEEE`, `#FFFFFF`
- [ ] **다크 모드 (필수, NFR-004)**: 라이트·다크 모드 모두 위 채도·그레이스케일 제약 충족. 구현: `prefers-color-scheme` 기반 자동 적용 + 사용자 선택을 영구 저장하는 수동 토글(자동 우선 적용, 토글 시 수동 값 우선). 검증: 다크 모드에서 텍스트 대비율 ≥ 4.5:1, UI 컴포넌트 대비율 ≥ 3:1. 테스트·리뷰 시 NFR-004 식별자로 참조.

### 대비

- [ ] **텍스트 대비**: 본문·헤딩 텍스트 대비율 ≥ 4.5:1 (WCAG AA)
- [ ] **UI 컴포넌트 대비**: 버튼·폼·카드 등 UI 요소 대비율 ≥ 3:1
- [ ] **호버·포커스 링**: 포커스·호버 시 채도 색(파란/보라 등) 사용 금지 — HSL saturation ≤ 5% 유지

### 토큰

- [ ] **border-radius**: 고유 값 ≤ 3개 (예: `0`, `4px`, `8px`)
- [ ] **box-shadow**: 고유 shadow 토큰 ≤ 3개 (예: `none`, `sm`, `md`)

### 성능

- [ ] **CSS 번들 크기**: **기준** 현재: (측정 예정) kB gzipped. **범위** 앱 전용(커스텀) CSS만 측정·제한 대상이며, 서드파티/벤더 CSS는 제외. **목표** 유지(80KB 이하) — 80KB는 상한 제약. **CI** 초과 시 빌드 실패.

### Status / StatusBadge 구현 노트

- **그레이 스케일 상태 표시 시 필수**: 상태(성공/경고/에러 등)를 그레이 스케일로만 구분할 때는 FR-004에 따라 **색에만 의존하지 않고**, (1) 아이콘, (2) 텍스트 라벨, (3) 구조적 큐(테두리 스타일/패턴, 그림자, 투명도 등) 중 **최소 2가지를** 사용한다. NFR-001 및 WCAG 2.1 SC 1.4.1을 만족하면서 FR-004의 모노크롬 요구와 충돌하지 않는다.
